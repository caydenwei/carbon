1. Problem

Bulk CSV imports in Carbon were functional but limited:

  • Shallow data: Importing items only set name and basic fields. Supplier-part links, lead times, and purchasing parameters required manual entry afterwards.
  • Shallow partners: Importing customers or suppliers created the record but not their address, payment terms, shipping method, status, or type — all had to be set one-by-one.
  • Fragile matching: Re-importing a CSV after deleting and recreating entities broke because orphan external-ID mappings pointed at deleted rows, causing FK violations.
  • No template guidance: The download-template button produced a bare header row; users had to guess which columns were required and what values were valid.
  • Manual enum mapping: Users had to manually map every CSV value (like employee emails) to system IDs, even when the values matched exactly.
  • Malformed CSVs hard-failed: A single mismatched row width in the CSV (common in exports from other ERPs) aborted the entire import.

2. System Context

CSV imports flow through three layers:

ImportCSVModal (UI)
→
imports.models.ts (schemas)
→
import-csv edge fn (server)
Layer File Responsibility
UI ImportCSVModal/* File upload, column mapping, enum value mapping, submit
Schema imports.models.ts Defines which columns exist per entity type, their types, validation, and lookup fetchers
Server import-csv/index.ts Supabase edge function: parses CSV, resolves entity matching, upserts rows + side-tables in a transaction

The schema layer is the single source of truth for both the UI (which columns to show, which need enum mapping) and the server (which fields to expect). Adding a new column to an import means adding it in the schema, then handling it in the server handler.

5.2 — Smart Template Download (UI)

The "Download Template" button was moved into the upload step and now generates a hint row beneath the headers, telling users what each column expects:

apps/erp/app/components/ImportCSVModal/UploadCSV.tsx
+ const hints = fields.map((f) => {
+   const prefix = f.required ? "REQUIRED" : "optional";
+   if (f.type === "enum" && f.enumData?.options) {
+     return `${prefix} — one of: ${f.enumData.options.join(" | ")}`;
+   }
+   if (f.type === "boolean") return `${prefix} — true | false`;
+   if (f.type === "numeric") return `${prefix} — number`;
+   return prefix;
+ });
+ const csv = Papa.unparse({ fields: headers, data: [hints] });

A user downloading the "Parts" template now gets a CSV like:

REQUIRED, REQUIRED, optional — see your configured options, ..., optional, optional — number

5.3 — Auto-Match Enum Values (UI)

When the user reaches the enum-mapping step, the UI now auto-matches CSV values to system options by case-insensitive comparison. This eliminates tedious manual mapping when the CSV already uses correct names:

apps/erp/app/components/ImportCSVModal/FieldMappings.tsx
+ const autoMatchedRef = useRef(false);
+ useEffect(() => {
+   if (autoMatchedRef.current) return;
+   const lookup = new Map(
+     options.map((o) => [o.label.toLowerCase().trim(), o.value])
+   );
+   for (const csvValue of uniqueValues) {
+     if (mappings[csvValue]) continue;
+     const matched = lookup.get(csvValue.toLowerCase().trim());
+     if (matched) onEnumMappingChange(name, csvValue, matched);
+   }
+   autoMatchedRef.current = true;
+ }, [options, mappedColumn]);

The useRef guard ensures auto-matching runs exactly once, so user edits aren't overwritten on re-render.

5.4 — Employee Email Matching

The accountManagerId fetcher on both customer and supplier now returns email from the DB. The enum label uses item.email ?? item.name, so CSV values containing employee emails match automatically via the auto-match logic above.

5.6 — Two-Tier Entity Matching (Server)

The biggest robustness improvement. Previously, entities could only be matched by their CSV external ID. Now the server falls back to matching by name:

packages/database/supabase/functions/import-csv/index.ts
 // For each CSV row:
+ const existingId = externalIdMap[csvId] ?? nameMap.get(name.toLowerCase());
 
 if (existingId) {
   // UPDATE existing entity
 } else {
   // INSERT new entity
 }

When a name-based match is used, a new CSV mapping is written so subsequent imports use the fast external-ID path.

5.8 — Side-Table Writes (Server)

The biggest functional addition. After the parent entity upsert, dedicated helpers write related records in the same transaction:

Entity Type Side-Table Helper Function
Supplier address, supplierLocation, supplierPayment, supplierShipping writeSupplierExtensions()
Customer address, customerLocation, customerPayment writeCustomerExtensions()
Item (all 5) supplierPart writeSupplierPartLinks()
Item (all 5) itemReplenishment writeItemPurchasingLeadTimes()
Item (all 5) itemPlanning writeItemPlanningOrderMultiples()

All side-table writes use bulk preloading (e.g., preloadPrimaryLocationAddressIds()) to avoid N+1 queries inside the transaction loop.

5.9 — Empty String Nullification (Server)

A simple but important utility that converts empty CSV cells to NULL before database writes. Without this, empty strings would violate FK and enum constraints:

packages/database/supabase/functions/import-csv/index.ts
+ function nullifyEmptyStrings(obj: Record<string, unknown>) {
+   for (const [k, v] of Object.entries(obj)) {
+     if (v === "") obj[k] = undefined;
+   }
+   return obj;
+ }
